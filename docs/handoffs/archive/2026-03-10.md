- 决策 3: 不在全局继续 deny `group:fs` / `group:runtime`
- 原因: 官方工具策略顺序里，早期 deny 无法被后续 agent allow 反向恢复；若不先下放，builder 永远拿不到 sandbox 内文件/执行面
- 放弃方案: 保留全局 deny，再在 builder 上追加 `allow`

- 决策 4: 为 builder 单独显式设 `tools.exec.host=sandbox` 与 `tools.elevated.enabled=false`
- 原因: 当前全局 `tools.exec.host=gateway` 是为“未启用 sandbox 的审批链”服务；builder 若沿用该值，会出现“声明用了 sandbox，exec 实际仍走宿主机”的假边界。同时 elevated 是官方定义的 host exec 逃逸口，builder 必须先压死
- 放弃方案: 只写 `sandbox.mode=all`，不改 builder 的 exec/elevated

- 决策 5: 用 `openclaw sandbox explain --agent builder` 做边界回看并修正 deny 细节
- 原因: 初版 `group:sessions` deny 会把 `session_status` 一起压掉；通过 explain 看有效 allow/deny 后，已改为仅 deny 不需要的具体能力，保留 `session_status`
