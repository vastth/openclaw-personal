# OpenClaw 编码风格规则

适用范围：本仓库所有代码与配置变更（JSON5、JS、Shell、Markdown）。

## 文件与函数规模

- 单文件不超过 **400 行**（最大容忍 800 行），超出前先考虑拆分。
- 单函数 / 单脚本块不超过 **50 行**。
- 嵌套深度不超过 **4 层**（条件/循环/回调）。

## 不可变性原则

- 不直接修改传入的对象或配置结构，返回新副本。
- 避免全局状态副作用（尤其对 `openclaw.json`、`settings.json` 的写入操作）。

## 错误处理

- 每个外部调用（API、文件读写、Shell 命令）必须显式处理错误。
- 不允许静默失败（catch 里空 block）。
- 面向用户的错误信息要友好；日志中保留详细上下文。

## 输入验证

- 所有来自用户输入、外部 API 响应、配置文件的数据，在使用前先验证结构。
- 使用 Schema 或显式字段检查，优先在系统边界拦截问题。

## 命名规范

- 配置 key：`camelCase`（与现有 `openclaw.json` 保持一致）。
- 文件名：`kebab-case`，与用途对应，不使用缩写。
- 临时 / 调试变量：不允许提交（`tmp_*`, `debug_*`, `test123` 等）。

## 文档、注释与交互语言

- 面向当前项目用户的说明、文档、交接记录、验证报告默认优先使用中文。
- 代码注释默认优先使用中文，要求简洁、准确，不写无意义注释。
- 命令、路径、环境变量、配置 key、协议字段、第三方产品名与固定术语保留原文。
- 若用户明确要求英文，或上游规范要求保留英文，则按实际场景处理。

## 代码完成前自检

- [ ] 没有硬编码 API Key / Token / Secret（见 security 规则）
- [ ] 函数 ≤50 行，文件 ≤400 行
- [ ] 嵌套深度 ≤4 层
- [ ] 每个错误路径都有处理
- [ ] 没有注释掉的废弃代码块（删除而非注释）
- [ ] 新增配置字段有注释说明用途

## JSON5 配置文件专项

- 每个新增字段旁必须有行注释说明用途（参照 `openclaw.json` 现有风格）。
- 不在配置文件中嵌入逻辑或计算表达式。
- 结构变更后，同步更新相关文档（`docs/` 或 `README.md`）。
